1. Doctype

We're using the quickly-becoming-ubiquitous HTML5 doctype, which is simple, clean and easy to remember. Not much to talk about here. It doesn't matter if you write it in uppercase or lowercase.


<!doctype html>

2. IE <html> classes

This is a mashup of code from Paul Irish and 320 And Up. This code wraps the html tag in a series of conditional statements, each targeting a different version of Internet Explorer. Use to target different versions of IE with classes in your CSS, as well as hook into the very handy .no-js class. 


<!--[if IEMobile 7]><html class="no-js iem7 oldie" lang="en"><![endif]-->
<!--[if lt IE 7]><html class="no-js lt-ie9 lt-ie8 lt-ie7" lang="en"> <![endif]-->
<!--[if (IE 7)&!(IEMobile)]><html class="no-js ie7 lt-ie9 lt-ie8 oldie" lang="en"><![endif]-->
<!--[if (IE 8)&!(IEMobile)]><html class="no-js ie8 lt-ie9 oldie" lang="en"><![endif]-->
<!--[if (gte IE 8)|(gt IEMobile 7)|!(IEMobile)]><!--><html class="no-js" lang="en"><!--<![endif]-->

3. a) Character-encoding declaration

The very first thing after you open your <head> tag should be the charset declaration. The whole tag has to be present within the first 1024 bytes of the page. Not declaring a charset isn't the end of the world, but it is best practice to do so, and you'll probably avoid a lot of strange errors by just declaring it and forgetting about it. 


<head>
    <meta charset="utf-8">

3. b) IE rendering options

These two lines address how your page renders in Internet Explorer. Courtesy of the HTML5 Boilerplate, the meta tag tells the IE rendering engine two things: first, it should use the latest, or edge, version of the IE rendering environment; second, if already installed, it should use the Chrome frameâ€™s rendering engine. This meta tag ensures that anyone browsing your site in IE is treated to the best possible user experience that their browser can offer. see below for more about Chrome Frame. Source.


<meta http-equiv="X-UA-Compatible" content="IE=edge,chrome=1">

3. c) Prefetching

Another very useful snippet from the HTML5 Boilerplate, which says it best in their documentation: in short, DNS Prefetching is a method of informing the browser of domain names referenced on a site so that the client can resolve the DNS for those hosts, cache them, and when it comes time to use them, have a faster turn around on the request. Source.


<link rel="dns-prefetch" href="//ajax.googleapis.com">

In this example, index.html has a prefetch instruction set up for Google CDN, where jQuery is called from in the <body>. This should speed up that HTTP request.

3. d) Basic metadata

These tags can be considered "good SEO" practice. Search Engine Optimization isn't entirely snake-oil, there are lots of things Google suggests we do as designers/developers when building websites. The title tag contains the page's official title (no, really?), and displays it in the browser tab. The description tag is important for search engine results, and the author tag is so you can get some credit for your work.


<title>Web Professional's Primer</title>
<meta name="description" content="The Web Professional's Primer is part field guide,  part encyclopedia, and part toolkit: the beginnings of an ambitious project organizing the collective knowledge of the web community into a killer reference manual, educational hub and resource library.">
<meta name="author" content="Zachary Kain">

<link rel="canonical" href="/">

The link rel="canonical" tag is used on pages with long URLs generated by a database, or pages that you want search engines to ignore in favor of a more official url. You can remove this line if you don't need it. Source.

3. e) Mobile viewport settings

Browser windows on mobile devices behave differently than in standard browsing environments. The first value here sets the viewport (browser window) width to be equal to the device width, which should eliminate horizontal scrollbars and activate any small-scale media queries in the CSS. The second value, initial-scale=1.0, prevents any default scaling done by mobile browsers and ensures that the page is initially viewed at "100%". Much, much more information in the sources: Source Source.


<meta name="viewport" content="width=device-width, initial-scale=1.0">

3. f) HTML5 shiv/shim/polyfill

The html5shim/shiv is a small piece of javascript written by Remy Sharp to enable default styling of HTML5 elements (article, section, figure, etc) in older browsers and Internet Explorer >9. The shim/shiv (they're two names for exactly the same thing) is an essential piece of "fire and forget" code, but if you use a library like modernizr, take it out since modernizr has the shim built in. Source.

            
<!--[if lt IE 9]><script type="text/javascript" src="http://html5shiv.googlecode.com/svn/trunk/html5.js"></script><![endif]-->

3. g) Favicons & mobile app icons

This sequence of links ensures cross-device compatible favicons and bookmark/app icons on iOS and Android. You don't need to include all this code as most of the magic actually happens automatically - see "Look ma, no HTML!" at the source: Source.

            
<!-- favicon, 16px -->
    <link rel="shortcut icon" href="favicon.ico">    
<!-- retina iphone, 114px -->
    <link rel="apple-touch-icon-precomposed" sizes="114x114" href="apple-touch-icon-114x114-precomposed.png">
<!-- ipad, 72px -->
    <link rel="apple-touch-icon-precomposed" sizes="72x72" href="apple-touch-icon-72x72-precomposed.png">
<!-- old iphone & android, 57px (also put apple-touch-icon.png in root) -->
    <link rel="apple-touch-icon-precomposed" href="apple-touch-icon-precomposed.png">

3. h) Loading CSS

Fairly straightforward, but a couple things to note here. The HTML5 spec does not require the type attribute, as its default value is text/css. That being said, I haven't switched over to not using it yet; it is good practice to include it to avoid edge cases with older browsers.

            
<link rel="stylesheet/css" type="text/css" href="css/styles.css?v=2">

The non-standard ?v=2 on the end of the stylesheet url is a form of version control used for what the HTML5 Boilerplate calls "Cachebusting". Given that web sites and applications are increasingly making use of caching (both standard methods and with HTML5 Cache Manifest), versioning links to .css and .js files means whenever those links are changed to a new version, browsers will ignore their cached copies and download the new file, ensuring any updates are received. Source.

3. i) FOUT-busting

FOUT, or "Flash-of-Unstyled-Text", is a phenomenon associated with using @font-face or other webfont techniques, the term coined by Paul Irish to mean when content on the page has loaded before the custom font files and is displaying as the fallback font, possibly breaking a layout or design. This script hides all text on the page until the webfonts have loaded, then displays the text - with a built-in safety if it takes too long. The end result is something that feels faster and more professional, even if it might technically be slower. Source.

            
<script src="js/foutbgone.js"></script>
<script>fbg.hideFOUT('asap', 300);</script>

I don't use this FOUT-busting method anymore, because I usually use Typekit to serve my web fonts. Much finer control over FOUT and font display overall can be had by using their built in "font-events" and Joni Korpi's page-loading techniques.

3. j) </head>

There's nothing else in the </head> element by default, except a <script> include for modernizr. All other javascript is loaded after the main content, to improve load time. Loading javascript blocks the rest of the page from loading (it doesn't load in parallel or asynchronously), so unless the script is vital (as in the case of FOUT-busting or modernizr), load it in the <body>, as you'll see below.

            
</head>

4. <body>

All the included code inside the <body> tags goes right at the very end, just before the closing tag. This is to ensure that the page content actually loads before our heavy files and scripts load.

4. a) jQuery

This method for loading the jQuery library comes from the HTML5 Boilerplate. Retrieving the file from Google's servers is arguably faster than from our own (especially when prefetched, as seen above). There is also a fallback script in place if the browser can't reach Google's servers. Source.


<script src="//ajax.googleapis.com/ajax/libs/jquery/1.7.1/jquery.min.js"></script>
<script>window.jQuery || document.write('<script src="js/libs/jquery-1.7.1.min.js"><\/script>')</script>

4. b) Google Analytics

Rather than using the standard Google Analytics snippet to monitor pages, use Mathias Bynens' asynchronous snippet for faster page loading and general goodness. Source.

  
<script>
    var _gaq=[['_setAccount','UA-XXXXX-X'],['_trackPageview']];
    (function(d,t){var g=d.createElement(t),s=d.getElementsByTagName(t)[0];
    g.src=('https:'==location.protocol?'//ssl':'//www')+'.google-analytics.com/ga.js';
    s.parentNode.insertBefore(g,s)}(document,'script'));
</script>

4. c) Script loading

As hinted at a few times already, it is very beneficial to page-load times if scripts and plugins are called after the main content on the page, to avoid a bottleneck while javascript loads. Not only that, but grouping many plugins or script files into one single file (one for scripts and one for plugins, load plugins first) will minimize the amount of HTTP-requests and speed up load times. As well, the defer attribute on the script tags lets the browser know that it may wait until the page has otherwise loaded before attempting to load the script in question, garnering even further potential load-time improvements. Source.


<script defer src="js/scripts.js"></script>

4. d) Chrome Frame

It is now possible to prompt the installation of Google's Chrome Frame instead of simply hoping it is installed. This IE-conditional statement prompts users of Internet Explorer 7 and below to install it. Source.


<!--[if lt IE 7 ]>
    <script defer src="//ajax.googleapis.com/ajax/libs/chrome-frame/1.0.3/CFInstall.min.js"></script>
    <script defer>window.attachEvent('onload',function(){CFInstall.check({mode:'overlay'})})</script>
<![endif]-->
